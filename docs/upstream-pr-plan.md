### Contributor levels:
- **Contributors** (us): no special privileges, max 1 open PR
- **Collaborators (Triage)**: significant contributions, own parts of the code, review/maintain
- **Maintainers**: review and merge PRs after code owner approval

### Mandatory for new quantization types (ggml_type extension):
1. **GGUF model conversion** — convert a small model, upload to HuggingFace
2. **Perplexity vs FP16/BF16** — `llama-perplexity` on wikitext-2 (native precision baseline)
3. **Perplexity vs similar-size types** — compare to q4_0, q3_K, etc.
4. **KL divergence vs FP16/BF16** — `llama-perplexity --kl-divergence` (logit files: 11-37 GiB)
5. **KL divergence for similar-size types** — same comparison set
6. **CPU performance** — `llama-bench` on CPU only, compare to similar-size types
7. **Test cases** — add to `test-backend-ops` for backend consistency (requires 2+ backends)
8. **Run full CI locally** — `bash ./ci/run.sh ./tmp/results ./tmp/mnt`

### Process rules:
- **Initial PR: CPU support only.** Metal/CUDA in follow-up PRs.
- Max 1 open PR for new contributors
- Squash-merge by maintainers (`<module> : <commit title> (#<issue_number>)`)
- No third-party dependencies, extra files, extra headers
- Cross-compatibility across OSes and architectures
- Snake_case everywhere, 4-space indent, clang-format v15+
- Use `struct foo {}` not `typedef struct`
- Sized int types (`int32_t`) in public API
- Enum values: UPPERCASE, prefixed with enum name
- Naming: `<class>_<method>` where `<method>` is `<action>_<noun>`
- Longest common prefix naming (e.g., `number_small`, `number_big` not `small_number`, `big_number`)
- C/C++ filenames: lowercase with dashes, `.h` / `.c` / `.cpp`
- Tensors store data in row-major order (dim 0 = columns, 1 = rows, 2 = matrices)
- `ggml_mul_mat(ctx, A, B)` means C^T = A * B^T ⟺ C = B * A^T (unconventional!)

### Community context:

## Codex 5.4 Review Findings (Applied)

Codex reviewed the initial version of this plan and identified 6 critical issues:

1. **Quality not proven for exact submitted variant** — the PPL 165.6 crisis is still in our docs. Must prove the *exact* CPU code path being submitted is quality-stable.
2. **GGUF requirement may not apply to KV-cache-only types** — TurboQuant is a runtime cache type, not a model quantization format. Need to clarify with maintainers.
3. **Need Phase 0 (consensus) before Phase 1 (code)** — don't burn the one-PR slot on unresolved design questions. Post in Issue #20977 first.
4. **Maintainer sponsorship is a blocker, not a nice-to-have** — find a willing collaborator/maintainer to champion the PR before submitting.
5. **Commit structure too coarse** — split graph changes from type/plumbing. Every commit must be buildable.
6. **Rotation data headers should be generated, not static** — 12K+ lines of pre-computed data will get pushback. Generate deterministically in code.

All addressed in the revised plan below.

## The Problem: Our Branch Is Metal-First

Our 43 commits are structured around Metal GPU optimization. Upstream wants CPU-first. We can't just clean up the branch and PR it — we need to restructure.

**Additional problem (from Codex):** CPU and Metal may not implement the same algorithm variant. CPU uses dense rotation in C code, Metal uses WHT. These produce numerically different results. The submitted CPU path must be self-consistent and independently validated.

## Revised PR Strategy: 4-Phase Approach

### Phase 0: Community Consensus (BLOCKING — before any code)
**Goal:** Get maintainer buy-in on the design before writing a single line of PR code.

**Actions:**
1. Post in Issue #20977 with:
   - Exact algorithm variant (MSE-only PolarQuant, no QJL, block size 32, graph-side WHT rotation)
   - Block struct layout with byte-level detail
   - Benchmark table: PPL vs FP16, vs q8_0, vs q4_0, vs q3_K
   - KLD data
   - CPU `llama-bench` numbers
   - Open questions:
     - Does a KV-cache-only `ggml_type` require a GGUF model upload, or is a reproducible benchmark script sufficient?
     - Is deterministic rotation matrix generation acceptable, or do maintainers prefer a smaller lookup table?
     - Preferred naming: `turbo3_0` vs `tq3_0` vs something else?
   - Link to fork for anyone who wants to test
2. Identify a willing collaborator/maintainer sponsor
3. Wait for design feedback before proceeding

**Exit criteria:** At least one maintainer/collaborator has acknowledged the approach and not raised blockers.

### Phase 1: Minimal CPU Reference (PR #1)
**Goal:** Get `GGML_TYPE_TURBO3_0` accepted as a new KV cache type with CPU backend only.

**Algorithm variant:** Freeze on whatever is quality-proven on CPU. If block-32 + MSE-only + pre-rotate is validated on CPU with PPL confirmed, use that. If not, fall back to block-128 (last known-correct) and optimize in Phase 2.

**What goes in:**
- `ggml-common.h`: `block_turbo3_0` struct definition
- `ggml.h` / `ggml.c`: `GGML_TYPE_TURBO3_0` registration, type traits
- `ggml-turbo-quant.c`: CPU quantize/dequantize (MSE-only, no QJL)
- `ggml-cpu/ops.cpp`: CPU backend support for turbo3
- `src/llama-kv-cache.cpp`: KV cache allocation for turbo3
- `src/llama-graph.cpp`: Graph-side WHT rotation (pre-rotate-queries)
- `common/arg.cpp`: `--cache-type-k turbo3 --cache-type-v turbo3` CLI flags
- `tests/test-backend-ops.cpp`: Backend consistency tests for turbo3
- Rotation matrix generation code (deterministic, NOT static headers)

**What stays out:**
- All Metal kernel code
- All Metal-specific headers (turbo-wht.h, turbo-matrices.h)
- GGML_OP_TURBO_WHT
- Quality gate script
- All documentation commits
- Static rotation data headers (replaced by generation code)

**Commit structure (4 commits, each buildable):**

## Open Questions for Phase 0

1. Does a KV-cache-only `ggml_type` require a GGUF model upload? Or is a reproducible benchmark script + reference model sufficient?
2. Is deterministic rotation matrix generation (seeded PRNG → orthogonal matrix) acceptable? Or do maintainers prefer a different approach (e.g., Hadamard only, no random component)?
3. Preferred type naming: `GGML_TYPE_TURBO3_0` or `GGML_TYPE_TQ3_0` or other?
4. Is graph-side rotation (modifying `build_attn` to insert `ggml_mul_mat` for Q rotation) acceptable scope for a quantization PR? Or should it be a separate feature PR?
5. Would maintainers prefer the rotation to be inside the dequant path (simpler graph, slower) or graph-side (faster, but touches more files)?
